# Android Artifact Hosting

This folder defines the OTA-style artifact flow for:

- `apps/android_deployer`
- `apps/app_tester`

The chosen model is:

- heavy native iteration: install directly with `adb install -r`
- realistic user update flow: publish signed APKs plus `latest.json` manifests to the VPS over HTTPS, then let `android_deployer` fetch and install them

## Hosting Layout

Recommended VPS root:

```text
/var/www/netab/android/
```

Recommended public URLs:

```text
https://debian13.ispot.cc/netab/android/
https://debian13.ispot.cc/netab/android/index.json
https://debian13.ispot.cc/netab/android/android_deployer/stable/latest.json
https://debian13.ispot.cc/netab/android/app_tester/dev/latest.json
```

Per-channel layout:

```text
<root>/<app>/<channel>/latest.json
<root>/<app>/<channel>/builds/<app>-v<versionCode>-<gitShort>.apk
```

Example:

```text
/var/www/netab/android/app_tester/dev/latest.json
/var/www/netab/android/app_tester/dev/builds/app_tester-v42-25d2d07.apk
```

`app_tester` is no longer just a remote probe shell. Current APKs now include:

- a real embedded Android `coord` node
- local route-state persistence in on-device SQLite
- local saved-node and persistent-connect restore in that same SQLite DB
- a small local steng table/doc playground backed by SQLite
- the existing remote VPS control surface for `cord.foundation.*`

## Linux Publish Flow

If the Linux machine does not already have Java / Gradle / Android SDK installed system-wide, you can set up a rootless local toolchain under `/home/<user>/android`:

```bash
npm run android:prepare-local
source ~/.local/bin/netab-android-env
```

Then build an app directly from the repo:

```bash
npm run android:build:tester
npm run android:build:deployer
```

The expected debug APK outputs are:

```text
apps/app_tester/app/build/outputs/apk/debug/app-debug.apk
apps/android_deployer/app/build/outputs/apk/debug/app-debug.apk
```

If this VPS owns the live web root and your user can write to it, you can also do the whole cycle in one command:

```bash
npm run android:publish-live
```

That command:

- builds `android_deployer` and `app_tester`
- injects a fresh Android `versionCode`
- publishes straight into `/var/www/html/netab/android`
- refreshes `index.html`, `index.json`, and both channel manifests

If port `443` is not configured on the host yet, `android:publish-live` falls back to:

```text
http://debian13.ispot.cc/netab/android
```

Once HTTPS is configured, you can force the base URL explicitly:

```bash
NETAB_ANDROID_BASE_URL=https://debian13.ispot.cc/netab/android npm run android:publish-live
```

1. Build and sign the APK on a machine with Android Studio / Android SDK.
2. Stage the manifest locally with the repository script.
3. Upload the staged directory to the VPS.

Stage locally:

```bash
npm run android:publish -- \
  --app app_tester \
  --channel dev \
  --apk /path/to/app-debug.apk \
  --dest-root ./deploy/android/out \
  --base-url https://debian13.ispot.cc/netab/android \
  --package-name cc.ispot.netab.apptester \
  --version-code 42 \
  --version-name 0.1.0-dev.42 \
  --min-sdk 26
```

Upload to VPS:

```bash
npm run android:upload -- \
  --source-root ./deploy/android/out \
  --remote user@debian13.ispot.cc:/var/www/netab/android
```

The publish script writes:

- the APK into the correct `builds/` folder
- `latest.json` for the requested app/channel
- root `index.json` with discovered apps/channels
- root `index.html` so `https://debian13.ispot.cc/netab/android/` is a tablet-friendly install page

## ADB Flow

For heavy native development, use ADB directly:

```bash
adb install -r /path/to/app-debug.apk
```

This avoids the OTA loop while you are changing Android code rapidly.

## OTA Flow

For a more realistic end-user experience:

1. keep `android_deployer` installed on the Fire tablet
2. publish `app_tester` to the VPS
3. open `https://debian13.ispot.cc/netab/android/` in the browser for first install, then use `android_deployer`
4. check the `app_tester/dev` channel
5. tap update
6. confirm Android’s install prompt

That gives a realistic “device pulls its own update” workflow without ADB.

## VPS Nginx

Use the sample config in [`nginx.netab-android.conf`](/home/user/src/netab/deploy/android/nginx.netab-android.conf).

The important parts:

- serve `/netab/android/` over HTTPS
- keep JSON and APK files publicly readable
- disable aggressive caching on `latest.json`
- allow longer caching on versioned APK files

## Signing

Do not sign APKs on the VPS.

Recommended rule:

- build and sign on your development machine
- upload only finished signed artifacts plus manifests to the VPS

That keeps signing keys off the server.

## Android Studio Notes

This repository does not ship a Gradle wrapper yet because the current Linux workspace does not include Java/Gradle/Android SDK tools.

Use Android Studio to open each project under `apps/`, let it sync with the Android Gradle Plugin, and build APKs there. Once you have a preferred Android build machine, you can add `gradlew` later if you want fully scripted APK builds inside the repo.
